# 路由权限系统（RBAC）

RBAC 是 Role-Based Access Control 的缩写，中文意思是基于角色的访问控制。

它是一种广泛使用的访问控制模型，核心思想是：权限不直接授予用户，而是授予“角色”，再把“角色”分配给用户。

## 🚀 一、最终目标

中大型后台系统权限架构

### 🎯 后端存数据库：

- menus（菜单树）
- roles（角色）
- permissions（权限点）
- role_permissions（角色权限关系）
- users（用户）

### 🎯 前端能力：

- 根据接口生成菜单
- 根据权限控制页面
- 根据权限控制按钮
- 动态路由

## 🧱 二、数据库设计

权限关系是： 用户 → 角色 → 菜单

核心表

```text
users        ✔ 用户
roles        ✔ 角色
user_role    ✔ 用户-角色
menus        ✔ 菜单 + permission
role_menu    ✔ 角色-权限
```

1. 用户表

纯用户表（不承载角色关系）

```sql
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL COMMENT '用户名',
  email VARCHAR(100) UNIQUE COMMENT '邮箱',
  password VARCHAR(100) NOT NULL COMMENT '密码',
  age INT DEFAULT NULL,

  status TINYINT DEFAULT 1 COMMENT '1正常 0禁用',

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

2. 角色表

roles 只回答一个问题： 👉 “这个用户在系统里是什么权限等级？”

```sql
CREATE TABLE roles (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL COMMENT '角色名称',
  code VARCHAR(50) NOT NULL UNIQUE COMMENT '角色标识（系统使用）',
  level INT DEFAULT 3 COMMENT '权限等级：1最高，数字越大权限越低',

  description VARCHAR(200) DEFAULT NULL COMMENT '角色描述',
  status TINYINT DEFAULT 1 COMMENT '1启用 0禁用',

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

3. 用户-角色表

它只做一件事： 👉 建立“用户 ↔ 角色”的多对多关系

```sql
CREATE TABLE user_role (
  user_id INT NOT NULL COMMENT '用户ID',
  role_id INT NOT NULL COMMENT '角色ID',

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  PRIMARY KEY (user_id, role_id)
);
```

4. 菜单表

menus 不只是“菜单”，而是：👉 UI + 权限点 + API权限的统一载体

```sql
CREATE TABLE menus (
  id INT PRIMARY KEY AUTO_INCREMENT,

  name VARCHAR(50) NOT NULL COMMENT '菜单/按钮名称',
  username VARCHAR(50) UNIQUE,
  code VARCHAR(50) NOT NULL UNIQUE COMMENT '系统标识（权限判断用）',

  path VARCHAR(100) DEFAULT NULL COMMENT '前端路由',
  component VARCHAR(100) DEFAULT NULL COMMENT '前端组件路径',

  parent_id INT DEFAULT NULL COMMENT '父级菜单ID',

  type TINYINT NOT NULL DEFAULT 1 COMMENT '1菜单 2按钮 3API权限',

  permission VARCHAR(100) DEFAULT NULL COMMENT '权限标识（核心字段）',

  icon VARCHAR(50) DEFAULT NULL COMMENT '图标',

  sort INT DEFAULT 0 COMMENT '排序',

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

```text
仪表盘
系统管理
 ├─ 用户管理
 │   ├─ 用户新增
 │   ├─ 用户编辑
 │   ├─ 用户删除
 ├─ 角色管理
 │   ├─ 角色新增
 │   ├─ 角色删除
 ├─ 菜单管理
     ├─ 菜单新增
     ├─ 菜单删除
```

5. 角色-权限表

roles ↔ menus 绑定关系

```sql
CREATE TABLE role_menu (
  role_id INT NOT NULL,
  menu_id INT NOT NULL,
  PRIMARY KEY (role_id, menu_id)
);
```

## 🚀 三、后端结构（admin-server）

Node 后端里（尤其 MVC / 三层架构）： controller → service → model → db

🔵 Controller（接收请求）

🟡 Service（业务逻辑）

🟢 Model（只查数据库）

```text
apps/admin-server/src/
├─ db/
│  └─ index.ts
├─ models/
│  └─ user.model.ts
├─ controllers/
│  └─ user.controller.ts
├─ services/
│  └─ user.service.ts
├─ routes/
│  └─ user.routes.ts
```

### JWT 中间件 + 权限拦截

1. 自动解析 token（authMiddleware）
2. 注入 user 到 req
3. 接口级权限控制（permissionMiddleware）

   token → userId → 查权限 → 判断是否放行

### 性能 + 缓存优化

1. 登录时缓存 RBAC
2. 请求时直接读取缓存
3. 不再查 DB（或极少查）

## 🚀 六、前端结构（admin-web）

```text
Token（登录态）
   ↓
Auth State（用户信息）
   ↓
Permissions（权限码）
   ↓
Menus（UI展示）
   ↓
Routes（静态 + 权限过滤）
```

```text
src/
 ├── router/
 │    ├── index.tsx        # 固定路由入口（不依赖 store）
 │    ├── routes.ts        # 静态路由
 │    ├── guard.tsx        # 权限守卫
 │    ├── lazy.ts          # 懒加载封装
 │
 ├── auth/
 │    ├── store.ts         # token/user/permissions
 │    ├── permission.ts    # 权限判断工具
 │
 ├── layout/
 │    ├── BasicLayout.tsx
 │
 ├── menu/
 │    ├── service.ts       # menus接口
 │    ├── transform.ts     # menus -> UI菜单
 │
 ├── pages/
```

